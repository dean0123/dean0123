
# 公開頁面功能分析：容量規劃與效益評估

## Context

PenPage 目前是 local-first 架構，所有頁面資料存在使用者瀏覽器的 IndexedDB，雲端同步透過 Google Drive（使用者自己的帳號）。伺服器端只有輕量 API（feedback、analytics、share email、admin），資源限制為 256MB RAM + 0.3 CPU。

評估目標：讓每位使用者可上傳一定數量的 page 到 PenPage server 作為**公開頁面**的可行性。

---

## 1. 單頁大小估算

### 純文字內容

| 頁面類型 | Markdown 大小 | 說明 |
| --- | --- | --- |
| 短筆記 | 0.5 - 2 KB | 幾段文字 |
| 一般文章 | 5 - 15 KB | 1000-3000 字含格式 |
| 長文 | 20 - 50 KB | 深度文章、教學 |
| 極長文 | 50 - 200 KB | 書籍章節等級 |


**合理中位數：~10 KB/頁**

### 含圖片

目前圖片儲存在 IndexedDB，使用 `image://` scheme。如果公開頁面要支援圖片，需要額外考慮：

| 圖片情境 | 壓縮後大小 | 說明 |
| --- | --- | --- |
| 無圖 | 0 | 純文字 |
| 1-3 張圖 | 200KB - 1MB | 經壓縮（max 1024px, 85% quality） |
| 圖片密集 | 2 - 5MB | 多張截圖/照片 |


**含圖中位數：~500 KB/頁**（假設平均 2 張圖）

## 2. 容量規劃模型

### 場景 A：小規模（MVP）

| 參數 | 值 |
| --- | --- |
| 每人公開頁數上限 | **5 頁** |
| 活躍使用者數 | 100 人 |
| 每頁含圖平均大小 | 500 KB |
| **總儲存** | **100 × 5 × 500KB = 250 MB** |
| 純文字版（不含圖） | 100 × 5 × 10KB = **5 MB** |


### 場景 B：中規模

| 參數 | 值 |
| --- | --- |
| 每人公開頁數上限 | **10 頁** |
| 活躍使用者數 | 1,000 人 |
| 每頁含圖平均大小 | 500 KB |
| **總儲存** | **1,000 × 10 × 500KB = 5 GB** |
| 純文字版 | 1,000 × 10 × 10KB = **100 MB** |


### 場景 C：大規模

| 參數 | 值 |
| --- | --- |
| 每人公開頁數上限 | **20 頁** |
| 活躍使用者數 | 10,000 人 |
| 每頁含圖平均大小 | 500 KB |
| **總儲存** | **10,000 × 20 × 500KB = 100 GB** |
| 純文字版 | 10,000 × 20 × 10KB = **2 GB** |


### 伺服器規格建議

| 規模 | 磁碟 | RAM | CPU | 月費估算 (VPS) |
| --- | --- | --- | --- | --- |
| 小型 (MVP) | 20 GB SSD | 1 GB | 1 core | ~$5-6/月 |
| 中型 | 50 GB SSD | 2 GB | 2 cores | ~$12-18/月 |
| 大型 | 200 GB SSD | 4 GB | 2-4 cores | ~$24-48/月 |


> 目前 PenPage API 容器只用 256MB RAM，公開頁面功能會需要**升級伺服器資源**。

---

## 3. 關鍵設計決策

### 圖片策略（影響最大）

| 方案 | 儲存成本 | 複雜度 | 建議 |
| --- | --- | --- | --- |
| **A. 不支援圖片** | 極低（純文字 ~10KB/頁） | 低 | MVP 首選 |
| **B. 圖片存伺服器** | 高（~500KB/頁） | 中 | 成本快速增長 |
| **C. 圖片用外部連結** | 零（使用者自行貼 URL） | 低 | 折衷方案 |
| **D. 圖片上 CDN/S3** | 中（有優化空間） | 高 | 未來擴展 |


**建議 MVP 用方案 A 或 C**，大幅降低伺服器成本與複雜度。

### 限制策略

建議的配額設計：

- **每人上限**：5-10 頁
- **單頁大小上限**：100 KB（純 Markdown）或 2 MB（含圖）
- **總配額**：每人 10 MB
- **Rate Limit**：每分鐘 5 次發布操作

## 4. 效益評估

### 正面效益

| 效益 | 說明 |
| --- | --- |
| **SEO 與曝光** | 公開頁面可被搜尋引擎索引，帶來自然流量。這是 local-first app 最缺的增長管道 |
| **社群分享** | 使用者可分享筆記連結（目前只能透過 email），降低分享門檻 |
| **產品展示** | 公開頁面本身就是 PenPage 編輯能力的 showcase |
| **使用者黏著** | 「我的公開主頁」增加使用者對平台的投入感 |
| **網路效應** | 讀者看到好的筆記 → 想試用 PenPage → 新使用者 |
| **差異化** | 在 local-first 筆記工具中，「一鍵發布」是有價值的功能（類似 Notion 的 Share to web） |


### 風險與挑戰

| 風險 | 嚴重度 | 說明 |
| --- | --- | --- |
| **架構轉變** | 高 | PenPage 從「零伺服器儲存」變成「伺服器存使用者內容」，這是根本性的架構轉變 |
| **隱私敘事衝突** | 高 | PenPage 核心賣點是「隱私優先、資料在你手上」。伺服器存內容與此矛盾，需要謹慎定位 |
| **內容審核** | 中 | 公開頁面 = 你的伺服器託管使用者內容 → 可能有不當內容、版權問題、法律責任 |
| **伺服器成本** | 中 | 從幾乎零成本的靜態站變成需要持續增長的儲存 |
| **濫用風險** | 中 | 垃圾內容、SEO spam、釣魚頁面 |
| **維運負擔** | 中 | 備份、監控、清理、效能優化 |

## 5. 綜合建議

### 這個功能對 PenPage 有益，但需要策略性地實作

**核心價值**：公開頁面解決了 local-first app 最大的弱點——**缺乏增長飛輪**。目前 PenPage 的傳播只能靠口碑，公開頁面讓每個使用者都成為潛在的推廣者。

**建議的實作路徑**：

1. **Phase 1 (MVP)**：純 Markdown 公開頁面，不含圖片
    - 每人 3-5 頁上限
    - URL 格式：`penpage.com/p/{pageId}` 或 `penpage.com/@{username}/{slug}`
    - Server-side rendering 或靜態生成（利於 SEO）
    - 儲存：SQLite 足矣（跟 analytics 一樣）
    - 成本增量：幾乎為零
2. **Phase 2**：加入圖片支援
    - 圖片壓縮後存伺服器或 S3
    - 提高配額到 10 頁
3. **Phase 3**：使用者個人主頁
    - `penpage.com/@username` 列出所有公開頁面
    - 自訂主題/排版

### 關於隱私敘事的化解

定位為：**「你的資料預設是私有的，你選擇公開的才會上傳」**。這反而強化了隱私敘事——使用者有完全的控制權，而不是像其他平台預設把資料放在雲端。

## 6. 總結數字速查

| 問題 | 答案 |
| --- | --- |
| 一頁多大？ | 純文字 ~10KB，含圖 ~150KB（640px/75%） |
| MVP 需要多少空間？ | 100 人 × 5 頁 × 150KB = **75 MB** |
| 中期需要多少空間？ | 1000 人 × 10 頁 × 150KB = **1.5 GB** |
| 現有伺服器夠嗎？ | 1000 人用最小 VPS ($5-6/月) 即可 |
| 對 PenPage 有益嗎？ | **有益**，是最有效的有機增長引擎 |
| 最大風險？ | 隱私敘事衝突、內容審核責任 |

## 7. 擴充性分析

### Scale Up（垂直擴充）

最直接，也是前期最務實的路線：

| 階段 | 規格 | 可承載 | 月費 |
| --- | --- | --- | --- |
| 現在 | 1C/1G/20G | ~1,000 人 | $5-6 |
| 中期 | 2C/4G/80G | ~10,000 人 | $20-24 |
| 後期 | 4C/8G/200G | ~50,000 人 | $48-60 |


SQLite 單機讀取效能優異，公開頁面以讀取為主的場景，scale up 能撐到數萬人。

### Scale Out（水平擴充）

目前架構的瓶頸點與升級路徑：

| 元件 | 現狀 | Scale out 需要的改動 |
| --- | --- | --- |
| **SQLite** | 單檔案，不支援多 writer | 換 PostgreSQL 或 Turso（分散式 SQLite） |
| **圖片儲存** | 本機 `/app/data/` | 改用 S3/R2 物件儲存（見下方說明） |
| **Fastify API** | 單容器 | 無狀態化後可多副本 + Load Balancer |
| **Nginx** | 單機反向代理 | 前面加 Cloudflare 或 LB |


### 建議的演進路線

```
Phase 1: 單機 SQLite（現在 ~ 數千人）
  ↓ 瓶頸出現時
Phase 2: 圖片搬到 S3/R2（解儲存瓶頸，成本更低）
  ↓
Phase 3: SQLite → PostgreSQL（解寫入瓶頸）
  ↓
Phase 4: API 多副本 + LB（解運算瓶頸）
```

每一步都有明確的升級路徑，MVP 用 SQLite + 本機儲存不會走進死胡同。大多數獨立產品在 Phase 1-2 就能撐很久，到需要 Phase 3 時通常已有足夠收入支撐升級。

### S3/R2 物件儲存說明

物件儲存是專門存放大量非結構化檔案（圖片、影片、文件）的雲端服務，與本機儲存的差異：

|  | 本機儲存 (`/app/data/`) | 物件儲存 (S3/R2) |
| --- | --- | --- |
| 檔案放哪 | VPS 的硬碟 | 雲端分散式儲存 |
| 容量上限 | 受硬碟大小限制 | 幾乎無限 |
| 備份 | 自己處理 | 自動多副本冗餘 |
| 存取方式 | 檔案路徑 | HTTP URL |
| 成本 | 包在 VPS 月費裡 | 按用量付費 |


**S3 vs R2 比較**：

|  | AWS S3 | Cloudflare R2 |
| --- | --- | --- |
| 儲存費 | ~$0.023/GB/月 | ~$0.015/GB/月 |
| 下載流量 | **$0.09/GB**（貴） | **免費** |
| 免費額度 | 5 GB | 10 GB |


公開頁面的圖片是**讀取密集**（多人看、少人寫），R2 的免費流量優勢很大。以 1000 人 1.5 GB 圖片為例，R2 每月儲存費僅 **$0.02**，流量免費。

將圖片搬到 R2 可以把「儲存」跟「運算」解耦——VPS 只負責跑 API，圖片用 URL 直接存取，擴充時互不影響。

### 圖片壓縮參數更新（640px / 75% quality）

| 項目 | 舊設定 (1024px/85%) | 新設定 (640px/75%) |
| --- | --- | --- |
| 單張圖片平均 | ~150-200 KB | **~50-80 KB** |
| 一頁含 2 張圖 + 文字 | ~400-500 KB | **~120-170 KB** |
| 1000 人 × 5 頁 | ~2.5 GB | **~750 MB** |
| 1000 人 × 10 頁 | ~5 GB | **~1.5 GB** |
